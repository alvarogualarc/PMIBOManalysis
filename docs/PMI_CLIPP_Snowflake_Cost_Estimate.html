This estimate covers all Snowflake compute and storage costs for running the CLIPP BOM Analytics platform. Costs are broken down by component across four deployment scenarios. Enterprise edition is required for Row Access Policies and Fine-Grained Access Control (FGAC) replicating ARAS RLS.

Pricing assumptions: On-demand Enterprise pricing at $3.00/credit on AWS. Annual capacity commitment (pre-purchase) reduces all credit costs by 20–40%. PMI's existing Snowflake contract rate should be substituted for precise figures. All costs in USD.

1. SPCS — Application Hosting

The React + FastAPI application runs as a single Docker container in Snowflake Container Services (SPCS). The container serves the React build via nginx and exposes REST API endpoints via FastAPI. A STANDARD_1 node (4 vCPU, 16 GB RAM) is sufficient for the full stack at PMI's concurrency target (20 concurrent users).

SPCS charges 1 credit per STANDARD_1 node-hour. A second node is recommended in production for high availability.

2. Virtual Warehouse — Interactive Query Compute

BOM_WH serves all user-facing queries: BOM explosion, Where Used, BOM comparison, impact analysis, and the 19 BRD business questions. The warehouse auto-suspends after 60 seconds of inactivity. Cost is driven by the number of distinct query bursts, not by concurrent user count.

Typical pattern: user navigates to a page → 1–3 API calls fire → warehouse resumes → queries complete in <1 second → warehouse suspends 60 seconds later. Effective active time per user interaction: ~65 seconds.

  • Estimated 300 user-initiated query bursts per day (20 concurrent users, ~15 bursts per user)
  • Effective active warehouse time: ~30 minutes per business day
  • Monthly active time: ~11 hours (22 business days × 30 min)

3. ETL — Daily ARAS Data Load

Each day, a delta extract from ARAS PLM is loaded into Snowflake staging tables. Load volume depends on how many Product Variants and BOM records changed since the previous extract. Two approaches exist: delta load (only changed records) and full reload.

3a. Staging Load Warehouse

A dedicated LOAD_WH (separate from BOM_WH) runs the COPY INTO and transformation steps. Running a Small warehouse for the duration of the load ensures the interactive query warehouse is unaffected.

After staging tables are updated, the BOM_CLOSURE table is rebuilt via recursive CTE. This runs on BOM_WH immediately after the delta load completes. The rebuild is the most compute-intensive single operation but is very short at tobacco BOM scale.

8. Assumptions and Notes

  • All prices based on Snowflake Enterprise edition, AWS US-East-1, on-demand pricing at approximately $3.00/credit.
  • Annual capacity commitment (pre-purchase) typically reduces credit costs by 20–40%. For a committed annual deployment, apply the negotiated rate to credit totals above.
  • SPCS STANDARD_1 node = 1 Snowflake credit per node-hour. Node spec: 4 vCPU, 16 GB RAM.
  • Warehouse active time estimates assume 60-second auto-suspend. Adjusting to 120-second suspend increases warehouse cost by ~50%; reducing to 30 seconds decreases it by ~25%.
  • Storage costs include 90-day Time Travel retention (3× compression factor). Fail-safe storage (7 days) adds a further ~10%.
  • ETL pipeline assumes ARAS provides a daily delta extract file. If ARAS only supports full exports, the weekly full reload figure applies daily, increasing ETL cost to ~$120–150/month.
  • These estimates do not include Snowflake platform fees, support tier costs, or professional services for implementation.
